nereid-server · build guide

From clone to a running server, in one script.

nereid links libtorch — a ~200 MB C++ dependency with a strict version pin and no embedded rpath. build.sh turns that from a research project into a single command, whether you're on a CPU laptop or a CUDA node behind a module system.

01 The short version

If you just want a working server and don't care how, this is the whole story.

bash
# CPU build. Downloads libtorch, builds, tells you how to run it.
$ ./build.sh

# Something you can copy to another box and run as-is — no LD_LIBRARY_PATH.
$ ./build.sh --release --link bundled --fetch-libtorch

The first command builds against the libtorch tch fetches for you and writes a launcher that sets the loader path. The second produces a self-contained directory under dist/ — binary plus its libtorch, wired to find each other wherever they land. That's the one you hand to a container or a colleague.

Why a script at all

libtorch ships as shared libraries with no rpath baked in, so a plain cargo build produces a binary that can't find its own dependency at runtime. Every mode below exists to solve that cleanly instead of exporting LD_LIBRARY_PATH by hand each time.

03 Prerequisites

The floor is a Rust toolchain. The rest depends on how far you go.

  • Rust (stable) + Cargo — the only hard requirement for a dynamic build. rustup.
  • curl + unzip — for --fetch-libtorch (present on essentially every Linux box).
  • patchelf — for --link bundled to set the rpath. Without it you still get a working bundle, just with a small run.sh wrapper instead of a bare binary. apt-get install patchelf.
  • A CUDA toolkit — only for --device cuda. On a cluster this is a module load away; see §06.
  • git, CMake, Ninja, and a C++17 compiler — only for --build-libtorch (§07), plus a Python with PyYAML for PyTorch's code generators.
Nothing to install for the common case

A first-time ./build.sh on a stock machine needs only Rust. libtorch is downloaded on demand, and the build tells you exactly how to launch the result.

04 Linking modes

How the binary connects to libtorch — and therefore how portable the result is.

dynamic

default · fastest turnaround

Links libtorch's .so files as usual. Portable to nowhere on its own, but perfect while you iterate.

bundled

relocatable · ship it

Copies libtorch beside the binary, rpath set to $ORIGIN/lib. Runs anywhere as a unit.

static

one file · self-contained

libtorch linked into the binary. No .so to carry — but you must build libtorch yourself first.

dynamic — the default

An ordinary cargo build, plus one thing cargo won't do for you: after the build, the script finds the exact libtorch lib/ directory it linked and writes a launcher next to the binary.

bash
$ ./build.sh --release
==> Built: target/release/grpc-test
==> Run wrapper written: target/release/run-grpc-test.sh

$ target/release/run-grpc-test.sh   # loader path already set

bundled — the one you ship

Builds, then assembles a directory that carries its own libtorch and knows how to find it. Because the rpath is $ORIGIN-relative, you can move, copy, or tar the whole folder and it still runs — no environment variables, no system libtorch.

bash
$ ./build.sh --release --link bundled --fetch-libtorch
==> Copied 12 shared object(s) and the binary into dist/grpc-test
==> Patched rpath to $ORIGIN/lib — the bundle is relocatable.

$ ls dist/grpc-test
grpc-test  lib/
$ ./dist/grpc-test/grpc-test        # no LD_LIBRARY_PATH, from anywhere
Verified relocatable

The rpath is relative to the binary, not an absolute path — copy dist/grpc-test to another host, a scratch dir, or a container image and it resolves libtorch from its own lib/ every time.

static — one binary, nothing beside it

Sets LIBTORCH_STATIC=1 so libtorch is linked directly into the executable. The catch is upstream: PyTorch stopped publishing prebuilt static archives after 2.1.2, so there's no libtorch.a to download at the 2.5.1 version tch needs. Pair this with --build-libtorch (§07), which produces one from source. The script checks the archives are present before linking and confirms the finished binary has no libtorch dependency left.

05 Where libtorch comes from

Four sources, checked in order. The first one that applies wins.

Verified downloads with --fetch-libtorch

By default tch downloads libtorch during the build with no integrity check and drops it somewhere under target/. --fetch-libtorch takes that over: it pulls the official archive, checks it against a pinned sha256, and caches it under .cache/libtorch/ so repeat builds are instant and provenance is known.

bash
$ ./build.sh --fetch-libtorch
==> downloading libtorch: .../libtorch-cxx11-abi-shared-with-deps-2.5.1+cpu.zip
==> sha256 verified: 618ca54e…562fa
==> extracting into .cache/libtorch/cpu-2.5.1

The CPU 2.5.1 hash ships pinned. For another device or version, pass the expected digest with --libtorch-sha256 <hex> — or --no-verify to skip the check, if you know what you're doing.

An existing install with --libtorch

On a shared system libtorch is often already unpacked somewhere central. Point at it and the script uses it verbatim — this is also how you supply a specific CUDA build.

bash
$ ./build.sh --libtorch /opt/libtorch --device cuda
Version & ABI must match

An external libtorch has to be the version tch expects (2.5.1 for tch 0.18) and the cxx11 ABI. A mismatch shows up as link or load-time symbol errors, not a friendly message.

06 HPC & managed environments

The messy parts of a cluster build — modules, CUDA, and someone else's Python — handled inline.

This script started life as a one-cluster recipe: module load cuda, point LIBTORCH at a shared install, build. That pattern is now first-class and portable — the same flags work on any Lmod or environment-modules system.

login-node ~ $
$ ./build.sh --module cuda/12.6.0 --device cuda \
             --libtorch /depot/group/torch/libtorch --release

--module is repeatable — pass it once per module you need loaded before the compiler and CUDA are on PATH.

Building inside conda, pyenv, or a venv

Clusters rarely give you the Python you want by default, and the from-source libtorch build (§07) needs one with specific packages. Activate the right environment as part of the build instead of remembering to do it first:

bash
$ ./build.sh --conda nereid --release
$ ./build.sh --pyenv 3.12.4 --build-libtorch
$ ./build.sh --venv ./.venv --link bundled

Each activates its environment in the build shell before anything compiles, so the toolchain, the Python code generators, and the model backends all see the same interpreter. If the manager or environment name is wrong, the build stops with a clear message rather than silently using the system Python.

Matches the client tooling

The Python client ships a conda_env.yml with tritonclient included, so the same conda environment can drive both a nereid-native and a Triton client against your freshly built server.

07 Static libtorch from source

The long road — worth it when you need a single binary with no runtime dependency on libtorch at all.

Since no prebuilt static libtorch exists at 2.5.1, scripts/build-libtorch.sh builds one. ./build.sh --build-libtorch calls it for you, then static-links against the result and drops back out to finish the server build. You can also run it standalone to produce a reusable libtorch.a tree.

bash
# End to end: build libtorch, then a static server binary.
$ ./build.sh --release --link static --build-libtorch

# Or build libtorch once, reuse it however you like.
$ scripts/build-libtorch.sh --version 2.5.1 --out /opt/libtorch-static
$ ./build.sh --release --link static --libtorch /opt/libtorch-static

Under the hood it follows PyTorch's own CMake recipe: a shallow, submodule-complete checkout at the tag, configured with BUILD_SHARED_LIBS=OFF into a clean install prefix, then cmake --build --target install. The output is exactly the include/ + lib/ layout LIBTORCH expects, and the script refuses to hand back a tree that didn't actually produce libtorch.a.

Set expectations

This is a full PyTorch C++ build: a multi-gigabyte checkout and a compile that runs from tens of minutes to well over an hour depending on the machine. It needs CMake, Ninja, a C++17 compiler, and a Python with PyYAML. Prerequisites are checked up front, so it fails fast if something's missing rather than an hour in.

Useful knobs: --jobs N to cap parallelism (a full-width libtorch build is memory-hungry), --device cuda for a CUDA-enabled static build, and --keep-src to keep the checkout around for a faster rebuild.

08 Running & troubleshooting

One error accounts for nearly every "it built but won't start" report.

the classic
error while loading shared libraries: libtorch_cpu.so:
cannot open shared object file: No such file or directory

This means the binary is dynamically linked and the loader can't find libtorch. It's expected when you run a dynamic binary directly instead of through its wrapper. Three ways out, cleanest first:

  1. Run the generated launcher: target/<profile>/run-grpc-test.sh. It sets the loader path for you.
  2. Build --link bundled and run dist/grpc-test/grpc-test — the problem can't occur, the path is baked in.
  3. Set it yourself: export LD_LIBRARY_PATH=<libtorch>/lib:$LD_LIBRARY_PATH.

Once it starts, the server binds to server.bind_addr from nereid.yaml and loads models from server.ml_backends_path. Copy nereid.yaml.example to nereid.yaml before the first run — the server won't start without it.

Add --run to any build to launch the server the moment it finishes, through the correct wrapper for the mode you chose.
